    Ссылки

    • ЛР3

    Цель

    Научиться упаковывать FastAPI приложение в Docker, интегрировать парсер данных с базой данных и вызывать парсер через API и очередь.

    Ход работы

    Подзадача 1: Упаковка FastAPI приложения, базы данных и парсера данных в Docker

    1. Реализован эндпоинт /parse в парсере, чтобы была возможность вызывать парсинг клиентским запросом.
    @app.post("/parse")
async def parse_users(
    session: AsyncSession = Depends(get_async_session),
    pages_to_parse: int = 1,
) -> list[dict]:
    results = await parse(pages_to_parse)
    all_users = []
    for _, users in results:
        if not users:
            continue
        for user in users:
            name = user.get("name") or ""
            name_parts = name.split(maxsplit=2)
            name_parts = name.split()
            last_name = name_parts[0] if len(name_parts) > 0 else ""
            first_name = name_parts[1] if len(name_parts) > 1 else ""
            await create_parsed_user(
                session,
                username=user.get("username") or "",
                first_name=first_name,
                last_name=last_name,
                email=user.get("email") or "",
            )
            all_users.append(user)
    if not all_users:
        raise HTTPException(status_code=404, detail="No users found")
    return all_users

    1. Для изоляции и удобства развертывания приложения созданы отдельные Dockerfile'ы для FastAPI-приложения и парсера. В Dockerfile'ах устанавливаются все необходимые зависимости, копируется исходный код, настраивается рабочая директория и указывается команда запуска приложения.

    2. В Docker Compose файле описаны сервисы для FastAPI-приложения, парсера, базы данных Postgres. Для базы данных настроен volume для сохранения данных между перезапусками контейнеров. Все сервисы объединены в одну сеть для возможности взаимодействия друг с другом по внутренним именам.

    Подзадача 2: Вызов парсера из FastAPI

    1. В FastAPI-приложении реализован reverse proxy эндпоинт, который принимает запросы и пересылает их на сервис парсера. Для этого используется библиотека httpx. Эндпоинт принимает параметры запроса, тело и заголовки, и отправляет их на парсер, возвращая клиенту ответ парсера без изменений. Это позволяет вызывать парсер через основной API, не зная его внутреннего адреса.
    @router.post("/")
async def proxy_parse(
    request: Request,
    pages_to_parse: int = 1,
):
    async with httpx.AsyncClient(timeout=None) as client:
        body = await request.body()
        headers = dict(request.headers)
        resp = await client.post(
            PARSER_URL,
            content=body,
            headers=headers,
            params={"pages_to_parse": pages_to_parse},
        )
        return Response(
            content=resp.content, status_code=resp.status_code, headers=resp.headers
        )

    Подзадача 3: Вызов парсера из FastAPI через очередь

    1. Для асинхронной обработки задач установлен Celery и Redis (брокер и backend для хранения результатов). Celery позволяет запускать задачи в фоне, не блокируя основной поток приложения.

    2. В проекте создана Celery-задача, которая запускает парсинг и сохранение пользователей в базу данных. Задача реализована как отдельная функция, которая импортирует необходимые модули и работает с асинхронной сессией SQLAlchemy.

    3. В парсер добавлены эндпоинты для постановки задачи в очередь (/parse/async) и для проверки статуса задачи по task_id (/parse/async/status/{task_id}). Аналогичные reverse proxy эндпоинты реализованы в основном FastAPI-приложении, чтобы клиент мог работать только с одним API.

    4. В Docker Compose добавлены сервисы для Redis и Celery worker. Celery worker запускается в отдельном контейнере и слушает очередь задач, обрабатывая их независимо от основного приложения.
